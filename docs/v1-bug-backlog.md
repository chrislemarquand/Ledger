# v1.0.0 Bug Backlog

## Severity rubric
- `S0`: data loss, crash, or corruption.
- `S1`: broken core workflow.
- `S2`: polish and usability issues.

## Open issues
- [x] `S1` Verify context-menu parity and enabled states in list/gallery/menu bar across mixed selections: verified previously.
- [x] `S1` Trace and eliminate intermittent `Publishing changes from within view updates is not allowed` warning: cannot reproduce after AppKit/SwiftUI rewrites. Closed as fixed by those earlier changes.
- [x] `S2` Inspector picker fault `Picker: the selection "" is invalid and does not have an associated tag`: fixed. The `tag("")` entry was only added conditionally when `currentValue.isEmpty`, so SwiftUI faulted when it validated the selection before the ViewBuilder content was fully evaluated. Fix: `tag("")` ("‚Äî" / "Multiple values") is now always the first item in the picker, unconditionally ‚Äî consistent with the preset editor's "Not Set" entry.
- [x] `S2` **B27** Stale sidebar entries not pruned on relaunch: fixed. Root cause: startup reconcile skips `fileExists` validation for Desktop/Downloads subfolders (to avoid TCC prompts), so deleted subfolders of those paths persisted. Fix: `loadFiles` now detects `NSFileNoSuchFileError`/`NSFileReadNoSuchFileError` after a failed enumeration and immediately removes the stale favourite or recent entry, persists, and clears selection to nil. Permission errors are intentionally not pruned.
- [x] `S2` **B28** Scroll-into-view on mode switch: fixed. `scrollSelectionIntoView` was calling `collectionView.layoutSubtreeIfNeeded()`, but on a list‚Üígallery switch the containing clip view's bounds hadn't updated yet so item frames were stale. Fix: call `scrollView.layoutSubtreeIfNeeded()` instead, which drives layout top-down through the full scroll view hierarchy before querying item attributes.
- [x] `S2` **B29** QuickLook panel height inconsistency: fixed. `lockedHeight` was only cleared when the panel was not visible; re-opening while visible used a stale height. Fix: always clear `lockedHeight` in `present()`.
- [x] `S2` **B30** QuickLook re-centring after drag inconsistent: fixed alongside B29. The `panelDidResize` height-lock + re-centre handler is correct and necessary ‚Äî without it, QL anchors its bottom-left corner on resize (jumping left/up) and progressively shrinks the panel by using it as a bounding box for subsequent images.
- [x] `S2` **B40** QuickLook Reduce Motion: closed as framework-constrained. QLPreviewPanel owns its animation engine; Finder itself is unchanged with Reduce Motion on.
- [x] `S1` **B33** `EXIF:DateTimeDigitized` not writable: fixed. ExifTool names EXIF tag 0x9004 as `CreateDate`, not `DateTimeDigitized` ‚Äî `-EXIF:DateTimeDigitized=...` silently fails while `-EXIF:CreateDate=...` works. The read-side `candidateKeys` fallback already listed `CreateDate` (so display was partially working), but the `EditableTag` field definition still had key `"DateTimeDigitized"`, which drove the write command. Fix: change the field key to `"CreateDate"`. Both write and clear now work correctly.
- [x] `S2` **B31** View ‚Üí Sort By menu order: fixed. Reordered to Name, Date Created, Size, Kind and renumbered key equivalents ‚åò‚åÉ‚å•1‚Äì4 to match toolbar and list columns.
- [x] `S0` **B35** Partial apply failure is silent: fixed. Root cause: when a file has mixed pending edits (some tags writable, one not), exiftool writes the good tags and exits **0** ‚Äî so `run` returned success and the failure was silently swallowed. Only a write with an exclusively-bad tag exits 1. Fix: after the exit-code check in `ExifToolService.run`, scan stderr for exiftool's "doesn't exist or isn't writable" warning on write operations and throw even on exit 0. The file then goes to `failed` and the partial-failure count appears in the subtitle.
- [x] `S2` **B34** Apply success subtitle: fixed. Now shows "Applied N image(s)" using `result.succeeded.count`.
- [x] `S2` **B36** Undo/redo character-by-character: fixed. `updateValue` now coalesces within an edit session via `undoCoalescingTagID`; only the first keystroke in a field pushes an undo entry. InspectorView calls `endUndoCoalescing()` on focus change and selection change.
- [x] `S1` **B37** "Restore from Backup" stays enabled after restore: fixed. After a successful restore, operation IDs were never removed from `lastOperationIDs`/`lastOperationFilesByID`, so `hasRestorableBackup` kept returning `true`. Fix: remove each successfully-restored operation ID from both registries inside the restore Task.
- [x] `S2` **B38** Inspector section expand/collapse with Reduce Motion ‚Äî cannot reproduce. Works correctly. Closed as not a bug.
- [x] `S2` **B39** Sidebar section collapse/expand with Reduce Motion ‚Äî cannot reproduce. Works correctly. Closed as not a bug.
- [x] `S2` **B40** QuickLook open/close not simplified with Reduce Motion: closed as framework-constrained. See entry above.
- [x] `S2` **B41** Preset names not enforced as unique: fixed. Removed "Keep Both" / `saveAsDuplicate`. Alert now only offers "Replace" or "Cancel".
- [x] `S2` **B42** App writes through macOS advisory file locks silently: deferred to post-v1.0 (tracked as R22).
- [x] `S2` Monitor `CMPhotoJFIFUtilities err=-17102` and `IOSurface creation failed: e00002c2` log spam: not reproduced under heavy thumbnail load; no user-visible breakage observed. Closed as system-framework noise.
- [x] `S2` Validate reduced-motion behavior for inspector section expand/collapse and field focus scrolling: validated.
- [x] `S2` Full manual QA matrix execution and sign-off: complete for v1.0. Three items deferred to v1.0.1: partial restore failure subtitle, full status subtitle table pass, and chmod read-only apply test (grouped with B42 file-lock work).
- [x] `S1` Apply Metadata toolbar button not activating immediately after a metadata edit: fixed. `installUIRefreshObservers()` was missing `model.$inspectorRefreshRevision` and `model.$stagedOpsDisplayToken` from its observer list. Menu items work because they pull state fresh via `menuWillOpen`; toolbar items require explicit pushing. Without these observers, `refreshFromModel()` (which sets `applyChangesItem?.isEnabled`) was never called after an edit ‚Äî only after a selection change. Fix: add both publishers to `installUIRefreshObservers()` in `MainContentView.swift`.

- [x] `S1` Inspector preview and gallery thumbnail disappear on rotate undo/redo: fixed. `applyPendingEditState` was calling `invalidateInspectorPreviews`, clearing `inspectorPreviewImages[fileURL]` even though the disk file hadn't changed. The cached image is the raw disk image; the rotation is applied as a software transform on the fly via `displayImageForCurrentStagedState`. Fix: remove the `invalidateInspectorPreviews` call from `applyPendingEditState`; bump `stagedOpsDisplayToken` instead so gallery cells reconfigure with the updated transform, and call `recalculateInspectorState(forceNotify: true)` for an immediate re-render.

## Closed during this pass
- [x] `S1` Folder-switch UX regression: fixed. Two-part fix: (1) `BrowserView.body` restructured so `browserContent` is always the root view with overlays applied via `.overlay()`, eliminating the structural identity changes that destroyed/recreated AppKit VCs on every overlay transition; (2) `selectSidebar` sets `isFolderContentLoading = true` and defers `loadFiles` to the next task, giving SwiftUI a render pass to show the loading skeleton before the gallery's `reloadData()` flash is visible. Root cause of previous fix failures: `defer` cleared the flag before SwiftUI rendered (fix 1), and async minimum-delay kept the flag true but the structural identity change destroyed AppKit VCs causing a different flash (fix 2).
- [x] `S2` Gallery thumbnail corner-style regression: resolved. Thumbnail shape remains consistent from first paint through pending-edit dot appearance; no corner-style transition between unedited/edited states.
- [x] `S2` Inspector preview loading spinner alignment: resolved. Spinner is centered in the preview placeholder while loading.
- [x] `S1` Gallery zoom keyboard shortcuts with inspector focus: resolved. `‚åò+` and `‚åò‚àí` now trigger zoom in gallery mode even when focus is inside inspector controls.
- [x] `S1` Remove actor isolation warning for split resize observer callback (`MainContentView.swift`).
- [x] `S1` `NSHostingView is being laid out reentrantly` fault from inspector: fixed. Three synchronous @Published/AppKit mutations during SwiftUI update phases: (1) `TextField` binding `set` calling `model.updateValue` synchronously ‚Äî now deferred via `DispatchQueue.main.async`, consistent with all other inspector field types; (2) `.onChange(of: model.selectedFileURLs)` setting `focusedTagID = nil` synchronously ‚Äî setting `@FocusState` during the update phase triggers an AppKit first-responder change that calls `layout()` on the `NSHostingView` mid-render, now deferred; (3) `InspectorLocationMapView.updateNSView` calling `addAnnotation`/`setRegion` synchronously ‚Äî these propagate `setNeedsLayout` up to the `NSHostingView`, now deferred via `DispatchQueue.main.async { [weak view] in ‚Ä¶ }`.
- [x] `S1` **B24** Gallery zoom display regression: fixed. `model.$galleryGridLevel` was missing from `installRenderObservers()` in the browser container VC ‚Äî zoom changes never triggered `render()` and therefore never reached `applyColumnCount()`. Fix: added `observe(model.$galleryGridLevel)` to the observer list.
- [x] `S2` **B32** Gallery thumbnail pending-edit dot does not refresh after editing metadata. Fixed alongside B24: `model.$inspectorRefreshRevision` was also missing from `installRenderObservers()`. `updateValue()` bumps `inspectorRefreshRevision` on every staged edit; adding its observer triggers `render()` ‚Üí `renderState()` ‚Üí pending-dot update without waiting for a selection change.
- [x] `S1` **B25** Sidebar/browser mismatch after pin/unpin: fixed (two-part). (1) `pinFavorite` always passed `preferredSelectionID: id` to `refreshSidebarItems`, silently jumping selection to the newly-pinned item even when something else was selected ‚Äî and no `loadFiles` was ever called. Fix: only follow the selection to the new `"favorite::"` ID if the pinned folder was already selected; otherwise hold the current selection. (2) `unpinSidebarItem` only called `selectSidebar` when `wasSelected`; with no neighbour it fell through to `sidebarItems.first` (Desktop). Fix: pass `selectFirstWhenMissing: false`, prefer adjacent favourite ‚Üí restored Recent ‚Üí `nil` (empty state) for the landing selection, and always call `selectSidebar` when the selection changes so browser content is guaranteed to match the sidebar. Also fixed: unpin now restores the folder to Recents (pinning had stripped it from `locationItems` via `removeRecentLocation`, and unpin never re-added it).
- [x] `S1` **B26** List view selection grey/inactive after gallery‚Üílist mode switch: fixed. `makeFirstResponder(tableView)` must be called after the list view is visible ‚Äî calling it from the notification observer (before `render()` shows the view) failed silently because AppKit rejects focus on views inside hidden containers. Fix: track `lastRenderedViewMode` in the list VC; in `update()` (which runs after `applyBrowserModeIfNeeded()` has already shown the list), detect `justBecameActive` and call `focusListForKeyboardNavigation()` then.

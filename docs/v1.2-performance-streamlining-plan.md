# v1.2 Performance Streamlining Plan

Last updated: 2026-04-04

Purpose: reduce runtime memory pressure, binary size, and compile-time complexity for Ledger/SharedUI without removing user-facing features.

---

## Scope and Constraints

- Preserve all v1.2 features and workflows.
- Focus on high-ROI, low-regression changes first.
- Any aggressive pruning must include a smoke pass on import/apply/restore/rename/location/date-time flows.

---

## Baseline Snapshot (Current)

- Shipped app sample (`dist/Ledger-1.1.1.app`) is ~`48M`.
- Bundled ExifTool payload is ~`39M` of that app.
- Bundled Brotli test fixtures under ExifTool are ~`15M`.
- Release executable observed at ~`7.6M` before strip and ~`3.7M` after `strip -S -x`.
- Inspector preview cache remains high-cap (`maxInspectorPreviewCacheEntries = 600`) and preview preload can span whole folder sets.

---

Acceptance:
- Smaller release binary in app bundle.

## Phase 2: Runtime Memory and UI Work Reduction

### 2.1 Inspector preview cache policy

- [x] Reduce `maxInspectorPreviewCacheEntries` from current high cap to a bounded working set (target: `30-60`).
- [x] Restrict preview warmup to active selection + nearby items rather than whole-folder preloading.
- [x] Reduce broad preview churn where possible by limiting preload candidate scope to selection-neighborhood windows.

Acceptance:
- Lower peak resident memory in large folders.
- No visible regression in inspector preview responsiveness for normal navigation.

### 2.2 Thumbnail fallback behavior

- [x] Remove cache path that stores raw `NSImage(contentsOf:)` full-resolution fallbacks.
- [x] For decode fallback, downsample before memory-cache insertion.
- [x] Keep file-icon fallback for hard failures.

Acceptance:
- Cache memory use remains bounded and predictable on large/odd source files.
- No broken thumbnail display states.

Status notes (completed work):

- Inspector preview cache cap reduced from `600` -> `48`.
- Added selection-neighborhood preload targeting (radius/cap based) so deferred/background warm paths no longer bulk-warm whole folders.
- Replaced raw `NSImage(contentsOf:)` cache fallback path with ImageIO downsampled thumbnail fallback.

---

## Phase 3: CPU and Latency Cleanup

### 3.1 Batch rename preview efficiency

- [x] Ensure sort happens once (remove duplicate sorting across caller/service).
- [x] Reuse date formatter/parser instances in rename assessment paths.
- [x] Debounce preview recomputation for rapid token edits.

Acceptance:
- Less CPU churn while typing rename patterns on large folders.
- No changes to rename output determinism.

Status notes (completed work):

- Added `assumeSorted` fast path to `BatchRenameService` planning/assessment so pre-sorted caller paths avoid duplicate sort work.
- Added cached `DateFormatter` reuse for date output and EXIF date parsing in `BatchRenameService`.
- Added lightweight UI debounce for live preview recompute in `BatchRenameSheetView` (`120ms`).

---

## Phase 4: Anti-Bloat Architecture Guardrails (Post-RC Safe Window)

### 4.1 Large-file decomposition

- [ ] Split `MainContentView.swift` into feature-focused files (menu wiring, split-view shell, observers, context menu actions).
- [ ] Continue decomposition of large AppModel extensions where natural boundaries already exist.

### 4.2 State-surface reduction

- [ ] Audit `AppModel` `@Published` properties and isolate hot-path/transient state from broad UI observation where possible.
- [ ] Prefer narrower observable slices for browser/inspector/import status over one broad publish surface.

Acceptance:
- Smaller compile blast radius per change.
- Lower risk of accidental UI-wide refreshes from localized state changes.
